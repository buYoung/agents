# PDF 문서 작성 기준

## 1. 목적

이 문서는 사내 개발 문서형 PDF를 만들 때 재사용할 작성 기준을 정의한다.

목표는 다음과 같다.

1. PDF가 단순 출력물이 아니라 독자가 바로 이해하고 판단할 수 있는 설명 문서가 되게 한다.
2. 문서마다 제목, 목적, 용어, 사용 예시, 흐름, 의사결정, 검증 기준을 일관되게 담는다.
3. 한글 본문, 코드성 텍스트, 표, 코드 블록, 요청 예시 박스의 시각 스타일을 통일한다.
4. 생성 후 텍스트 추출과 렌더링 검수까지 완료한 PDF만 최종 산출물로 본다.

이 문서는 `output/pdf/`에 배치되는 설명용 PDF, 운영 가이드 PDF, 업무 흐름 설명 PDF에 적용한다.

## 2. 사실 기준

PDF 내용은 현재 코드와 문서의 실제 상태를 기준으로 작성한다.

우선순위는 다음과 같다.

1. 현재 저장소의 코드와 설정
2. 기존 스펙 문서와 FDD
3. 실제 실행 결과, 로그, 산출물
4. 사용자가 제공한 문서 목적과 피드백
5. 기존 PDF 초안

기존 PDF 문구는 신뢰 기준이 아니라 개선 대상이다. 이전 PDF에 적힌 내용이라도 현재 코드, 설정, 실행 결과와 다르면 현재 상태를 우선한다.

## 3. 문서 범위

PDF를 만들기 전에 아래 값을 먼저 확정한다.

| 항목 | 의미 |
| --- | --- |
| 문서 대상 | 설명할 제품, 플러그인, 기능, 실행 흐름 |
| 대상 독자 | 개발자, 운영자, 검토자, 의사결정자 중 누구인지 |
| 문서 목적 | 설치 안내, 동작 설명, 운영 판단, 검증 보고 중 무엇인지 |
| 포함 범위 | 반드시 설명할 기능, 설정, 산출물, 실행 방식 |
| 제외 범위 | 설치 절차, 내부 구현, 향후 계획처럼 이번 문서에서 다루지 않을 내용 |
| 근거 자료 | 코드 경로, 기존 문서, 실제 실행 결과, 사용자 피드백 |
| 최종 산출물 | PDF 경로와 필요하면 생성 스크립트 경로 |

범위가 불명확하면 PDF를 먼저 만들지 않는다. 독자와 목적이 모호한 PDF는 내용도 레이아웃도 쉽게 흔들린다.

## 4. 필수 내용 구조

설명용 PDF는 기본적으로 아래 순서를 따른다.

1. 제목과 부제목
2. 문서 목적
3. 용어 정리
4. 대상 개요
5. 설정 및 사용 예시
6. 실제 업무 기준의 동작 설명
7. 대표 사례
8. 호출 또는 실행 흐름
9. 의사결정 기준
10. 산출물 확인 기준
11. 결과 해석 기준
12. 최종 요약

모든 문서가 이 순서를 기계적으로 따라야 하는 것은 아니다. 다만 독자가 처음 읽어도 “무엇을 설명하는 문서인지”, “언제 쓰는지”, “어떻게 동작하는지”, “성공 여부를 어떻게 확인하는지”를 찾을 수 있어야 한다.

## 5. 제목과 식별자 기준

제목은 사람이 읽기 쉬운 이름을 사용한다. 패키지명, 경로, 명령어처럼 코드성 식별자는 부제목이나 본문 첫 문단에서 정확히 밝힌다.

권장:

- 제목: `hicare-agent opencode 플러그인 설명서`
- 부제목: `패키지명: @hicare-agent/opencode`

피해야 할 방식:

- 제목 전체를 패키지명만으로 구성한다.
- 프로젝트명과 패키지명을 섞어 쓴다.
- 오탈자가 있는 식별자를 그대로 둔다.

문서 첫 페이지에는 기준일과 설명 대상을 함께 적는다.

## 6. 내용 작성 기준

### 6.1 문서 목적

첫 페이지에서 이 문서가 무엇을 위한 문서인지 분명히 쓴다.

좋은 목적 문장은 다음 정보를 포함한다.

- 누가 읽는 문서인지
- 무엇을 이해해야 하는지
- 어떤 판단에 쓰는지
- 설치 절차인지, 동작 설명인지, 운영 판단 문서인지

예시:

```text
이 문서는 @hicare-agent/opencode 플러그인의 설치 절차보다,
플러그인이 제공하는 에이전트 구성, 호출 기준, 산출물 규칙,
실행 결과 해석 방법을 설명하는 데 초점을 둔다.
```

### 6.2 용어 정리

처음 등장하는 기술 용어는 초반에 표로 정리한다.

포함 대상:

- 제품 안에서 특별한 의미를 갖는 이름
- agent, hook, task처럼 일반 단어와 제품 의미가 다른 용어
- clean-run, MCP처럼 처음 읽는 사람이 문맥 없이 이해하기 어려운 용어
- 산출물 경로와 파일명

용어 설명은 내부 구현보다 독자가 판단에 필요한 의미를 우선한다.

### 6.3 설정 및 사용 예시

실제 설정 형식을 확정할 수 있으면 정확한 예시를 넣는다. 확정할 수 없으면 “이해를 돕기 위한 예시”라고 표시한다.

설정 예시는 코드 블록으로 분리한다.

```json
{
  "plugins": ["@hicare-agent/opencode"],
  "agent": {
    "default": "orchestrator"
  }
}
```

사용 예시는 실제 업무 요청 문장처럼 쓴다. 너무 인위적인 테스트 문구보다 독자가 복사해 변형할 수 있는 문장이 좋다.

### 6.4 흐름 설명

흐름 설명은 “무슨 일이 일어나는지”보다 “왜 그 단계가 필요한지”를 함께 보여준다.

각 단계는 아래 구조를 따른다.

| 항목 | 설명 |
| --- | --- |
| 입력 | 이 단계가 받는 요청이나 전제 |
| 행동 | 실제로 수행하는 일 |
| 의사결정 | 다음 단계로 넘길지, 멈출지, 직접 처리할지 판단하는 기준 |
| 산출물 | 파일, 요약, 검증 결과처럼 남기는 결과 |

의사결정은 문서 형식의 ADR을 그대로 넣지 않는다. 독자가 이해하기 쉬운 판단 문장으로 녹여 쓴다.

### 6.5 사례

사례는 실제 업무 기준으로 작성한다.

좋은 사례:

- 코드 권한 정책 개선
- 에이전트 호출 흐름 검증
- 특정 파일 수정과 좁은 검증
- 산출물 생성 여부 확인

피해야 할 사례:

- 모든 agent를 억지로 부르는 부자연스러운 문구
- 제품 의미와 무관한 장난감 예시
- 실제 사용자가 하지 않을 법한 테스트 전용 요청

### 6.6 산출물 확인 기준

PDF에는 성공 여부를 확인할 수 있는 기준을 반드시 둔다.

예시:

- 생성되어야 하는 파일 목록
- 각 파일에 들어가야 하는 최소 내용
- 실행 결과에서 확인해야 하는 호출 흔적
- 단독 실행과 전체 실행의 차이
- 실패로 봐야 하는 상태

## 7. 글꼴 기준

기본 폰트 조합은 다음을 권장한다.

| 영역 | 폰트 | 권장 굵기 |
| --- | --- | --- |
| 한글 본문 | Pretendard | Regular |
| 제목 | Pretendard | SemiBold 또는 Bold |
| 표 본문 | Pretendard | Regular |
| 코드, 파일명, 패키지명 | JetBrains Mono | Regular |
| 코드 강조 | JetBrains Mono | Medium |

권장 크기:

| 영역 | 크기 |
| --- | --- |
| 대제목 | 22~24pt |
| 섹션 제목 | 15~17pt |
| 본문 | 10~10.5pt |
| 표 본문 | 8.8~9.2pt |
| 코드 블록 | 9~9.5pt |
| 푸터 | 7pt |

한글 설명은 Pretendard로 두고, 기술 식별자만 JetBrains Mono로 처리한다.

## 8. 코드성 텍스트 기준

아래 항목은 일반 문장과 구분되게 코드 스타일로 표시한다.

- 패키지명: `@hicare-agent/opencode`
- agent 이름: `orchestrator`, `worker`, `research`
- 파일명: `work.md`, `adversarial-review.md`
- 경로: `.agents/{taskId}/work.md`
- JSON 키: `subagent_type`, `description`, `prompt`
- 코드 경로: `packages/opencode/src/core/permissions.ts`
- 명령어와 모델 이름

긴 파일명은 표 안에서 줄바꿈으로 깨지지 않게 열 폭을 조정한다. `constructive-feedback.md`처럼 의미가 파일명에 들어 있는 항목은 중간 줄바꿈이 생기면 안 된다.

## 9. 레이아웃 기준

### 9.1 페이지

- A4 기준을 기본으로 한다.
- 좌우 여백은 15~17mm 정도를 권장한다.
- 한 페이지에 너무 많은 표와 문장을 동시에 넣지 않는다.
- 페이지마다 하나의 중심 질문이 있어야 한다.

### 9.2 표

표는 정보 비교가 필요한 곳에만 쓴다.

권장:

- 헤더 색은 너무 진하지 않게 한다.
- 셀 내부 여백을 충분히 둔다.
- 표 본문은 9pt 전후를 유지한다.
- 행간은 1.25~1.35 수준으로 둔다.
- 코드성 텍스트는 JetBrains Mono로 처리한다.
- 긴 파일명과 경로는 줄바꿈이 깨지지 않게 열 폭을 조정한다.
- 필요하면 행 배경을 아주 연하게 교차 적용한다.

피해야 할 방식:

- 모든 설명을 표 안에 밀어 넣는다.
- 파일명 열이 좁아져 식별자가 깨진다.
- 표 글자가 8pt 아래로 내려가 읽기 어려워진다.

### 9.3 코드 블록

코드 블록은 본문과 명확히 구분한다.

권장:

- 배경: 연한 회색
- 테두리: 옅은 회색
- 폰트: JetBrains Mono
- 크기: 9~9.5pt
- 줄간격: 1.3 전후
- 패딩: 충분히

### 9.4 요청 예시 박스

요청 예시는 따로 눈에 띄는 박스로 만든다.

박스 안에서도 경로, 파일명, 명령어, agent 이름은 코드 스타일을 유지한다.

## 10. 생성 기준

PDF는 재현 가능하게 생성해야 한다.

권장 방식:

1. 생성 스크립트를 둔다.
2. 출력 경로는 `output/pdf/` 아래로 둔다.
3. 임시 렌더링 결과는 `tmp/pdfs/` 아래에 둔다.
4. 사용한 폰트 경로를 스크립트에 명시한다.
5. 표, 코드 블록, 흐름도는 재사용 가능한 함수로 분리한다.
6. 기준일은 자동 삽입하되, 문서 내용의 사실 기준은 현재 코드와 실행 결과로 검증한다.

생성 스크립트가 일회성이라도 다음 작업자가 수정할 수 있을 정도로 이름과 구조를 명확히 둔다.

## 11. 검증 기준

PDF 생성 후 아래 검증을 완료해야 한다.

### 11.1 텍스트 검증

PDF에서 텍스트를 추출해 다음을 확인한다.

- 페이지 수
- 제목과 부제목
- 필수 섹션명
- 필수 패키지명, 파일명, 경로
- 금지 문구와 오탈자 부재

예시:

```text
필수 문구:
- hicare-agent opencode 플러그인 설명서
- 패키지명: @hicare-agent/opencode
- 설정 및 사용 예시
- 용어 정리

금지 문구:
- openode
- 잘못된 패키지명
- 불필요한 내부 명칭
```

### 11.2 렌더링 검증

PDF 전체 페이지를 이미지로 렌더링한다.

확인 대상:

- 모든 페이지가 렌더링되는지
- 제목과 표가 잘리지 않는지
- 긴 파일명과 경로가 깨지지 않는지
- 코드 블록과 요청 예시 박스가 본문과 구분되는지
- 표 글자가 읽을 수 있는 크기인지
- 도형 안의 텍스트가 겹치거나 넘치지 않는지

렌더링 실패가 있으면 PDF 생성을 완료로 보지 않는다.

### 11.3 주요 페이지 확대 검수

전체 렌더링 후 핵심 페이지는 직접 확대해서 확인한다.

대상:

- 1페이지: 제목, 부제목, 문서 목적
- 용어 페이지: 코드성 용어 스타일
- 설정 페이지: 코드 블록
- 요청 예시 페이지: 박스와 코드성 텍스트
- 표 중심 페이지: 파일명 줄바꿈과 표 여백
- 흐름도 페이지: 도형 안 텍스트 겹침
- 최종 요약 페이지: 결론과 확인 기준

## 12. 완료 체크리스트

PDF 작업 완료 전 아래 항목을 확인한다.

```text
[ ] 문서 목적과 대상 독자가 첫 페이지에 드러난다.
[ ] 제목은 사람이 읽기 쉽고, 실제 패키지명은 부제목이나 본문에 정확히 표기됐다.
[ ] 용어 정리가 초반에 있다.
[ ] 설정 예시와 사용 예시가 있다.
[ ] 실제 업무 기준의 사례가 있다.
[ ] 동작 흐름에 의사결정 기준이 함께 설명된다.
[ ] 산출물 확인 기준이 있다.
[ ] 결과 해석 기준이 있다.
[ ] 한글 본문은 Pretendard 계열로 보인다.
[ ] 코드성 텍스트는 JetBrains Mono 계열로 보인다.
[ ] 코드 블록은 배경, 테두리, 패딩으로 분리된다.
[ ] 표 안의 긴 파일명과 경로가 깨지지 않는다.
[ ] PDF 텍스트 추출 검증을 통과했다.
[ ] 전체 페이지 렌더링 검증을 통과했다.
[ ] 주요 페이지 확대 검수를 통과했다.
[ ] 최종 PDF가 output/pdf/ 아래에 있다.
```

## 13. 재사용 절차

새 PDF를 만들 때는 이 문서를 먼저 읽고 다음 순서로 진행한다.

1. 문서 대상과 독자를 확정한다.
2. 현재 코드, 설정, 기존 문서, 실행 결과를 확인한다.
3. 필수 구조에 맞춰 목차를 잡는다.
4. 제목, 용어, 설정 예시, 사용 예시를 먼저 작성한다.
5. 실제 업무 사례와 의사결정 흐름을 추가한다.
6. 산출물과 결과 해석 기준을 넣는다.
7. 타이포그래피와 표 스타일을 적용한다.
8. PDF를 생성한다.
9. 텍스트 검증, 전체 렌더링, 주요 페이지 확대 검수를 수행한다.
10. 검증 결과와 최종 경로를 사용자에게 보고한다.

이 순서를 따르면 PDF가 단순한 화면 출력물이 아니라, 다음 사람이 같은 기준으로 다시 만들고 검토할 수 있는 재사용 가능한 문서가 된다.
